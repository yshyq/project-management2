# 项目优先的支持提交流程设计

## 目标

客户基础资料只在“客户信息”模块维护。项目部署负责创建客户对应的部署项目信息；技术支持、项目需求和其他支持直接选择已有部署项目，系统自动带出项目所属客户。若项目包含多个产品，提交人再选择本次支持涉及的具体产品。

## 业务规则

1. 项目部署：
   - 必须选择客户。
   - 手工填写项目名称。
   - 至少选择一个部署产品。
2. 非项目部署支持：
   - 不显示客户选择控件。
   - 从全部已部署项目中选择项目。
   - 选择项目后自动展示客户名称，并加载该项目的产品。
   - 项目有多个产品时必须选择一个具体产品。
3. 后端不信任前端提供的客户关系：
   - 非部署支持根据选中的部署项目确定 `customerId`。
   - 项目不存在或项目名称在多个客户下不唯一时拒绝提交。
   - 所选产品必须属于该部署项目。

## 接口设计

- `GET /api/deployment-project-options`
  - 不传 `customerId` 时返回全部部署项目。
  - 每项返回 `projectName`、`customerId`、`customerName`、`products`。
  - 保留可选 `customerId` 参数，兼容已有调用。
- `POST /api/support-tickets`
  - 项目部署继续接收 `customerId`、`projectName`、`productIds`。
  - 非部署支持以 `projectName` 和 `productId` 为业务输入；服务端从部署项目解析并写入客户。
  - 为兼容同名项目和现有请求结构，`customerId` 可作为项目消歧信息出现，但服务端必须验证它属于所选项目。

## 前端交互

- 打开非部署支持表单时并行加载全部部署项目。
- 第一项选择“项目”，选项展示“项目名称（客户名称）”。
- 选择项目后显示只读客户名称，并清空之前选择的产品。
- 产品下拉只展示该项目已部署产品。
- 切换支持类型或重新打开表单时清空项目、客户和产品状态。
- 加载失败、无部署项目、项目无产品均显示明确提示。

## 测试

- 后端接口测试覆盖全量项目选项、客户自动解析、客户错配修正、非法项目和非法产品。
- 前端单元与组件测试覆盖项目优先选择、客户自动带出、产品联动、校验与请求载荷。
- Playwright 覆盖客户创建、项目部署、三类非部署支持的完整浏览器闭环。
- 最终执行后端全量测试、前端单元测试、类型检查、生产构建和端到端测试。
